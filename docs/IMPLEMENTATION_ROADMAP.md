# Порядок реализации MiMusic (бэкенд + клиент, по этапам)

Цель — идти от **фундамента** к **фичам**, чтобы каждый этап давал рабочий срез и не блокировал следующий. Начало с **БД** — правильная интуиция: от схемы зависят API, загрузки файлов и права.

**Параллельно клиент и сервер:** после того как эндпоинты по этапу **стабильно** отвечают на dev, **в том же спринте/итерации** добавляйте на Flutter минимальную проверку в приложении (экран, список, настройка из `profile` / «Студии» или временный пункт меню). Не откладывайте весь UI до «одного большого этапа 12» — иначе расхождение бэка и того, что вы реально тестируете в APK, растёт. Оформление коммитов: `md/COMMIT_STYLE.md` (отдельные коммиты в `MiMusic/` и `mimusicback-master/`).

Подробная модель данных и файлов: `BACKEND_CLIENT_ALIGNMENT.md`. Краткий контекст: `AGENT_CONTEXT_BRIEF.md`.

**Статус:** ~~этап 0 (подготовка репозитория)~~ — **сделан** (env для БД и путей хранилищ, структура `file_storage`; при старте бэка подхватывается **`mimusicback-master/.env`** если переменные не заданы в ОС). ### Готовность экранов (Flutter UI)

Оценка **по UX и данным в приложении**, а не по наличию файла виджета: «готово» — основной сценарий стабилен end-to-end с API где нужно; «частично» — есть рабочий кусок, но неполная модель, локальный fallback или заготовки; «заглушки» — экран или вкладка в основном на моках / без реального бэка.

| Статус | Экраны и переходы |
|--------|-------------------|
| **Полностью готовы** | **Настройки** (общие); **Персональные настройки** — отступы шелла/мини-плеера, профиль где затронут. **Избранные (Любимые)** — лайки **`server_track_*`** ↔ **`track_likes`**, каталог `GET /tracks`. **Colisten** — ~~REST+WS, host/guest, права, открытые комнаты~~. **Поиск → люди** — `UsersApi.searchUsers`. |
| **Частично готовы** | **Поиск → музыка** — ~~треки с API~~; плейлисты публичные с API; альбомы в поиске — по желанию. **Профиль** — ~~`GET/PUT /me`, аватар~~; статистика треков/плейлистов/друзей — **заглушка** (п.12). **Плейлисты** — серверный CRUD и каталог. **Главная** — блок серверных треков внизу + плеер; релизы и рекомендательные ряды — моки (п.9–11). **Для вас / Чарты** — см. п.7–8. **Мысли** — ~~лента/создание/вложения, лайки, комментарии, аватарки, профиль по тапу~~. **Студия** — wizard upload на сервер; остальное локально (п.13). **Публичный профиль** — сервер; **ArtistPage** — локально (п.14). |
| **В основном заглушки / отстаёт** | **Друзья** — ~~mock без логина~~; для **залогиненного** пользователя — **`RemoteFriendsRepository`** / `FriendsApi`; остаётся полнота UX и сценариев. **Уведомления** — ~~чтение с API и экран~~ в all-in; полнота сценариев — по факту экрана. |

## Этап 13 — Финал диплома: рекомендательная аналитика и оценка

**Цель:** закрыть рекомендательную часть без тяжёлого ML, но с измеримым результатом для защиты.

- **Rule-based scoring:** зафиксировать формулу ранжирования (`genre preference` × `track genres` + простые бонусы/штрафы, если добавите) и сделать её единой для API рекомендаций.
- **Лог событий:** писать в `recommendation_events` минимум `impression`, `click`, `play_start`, `skip`; для показа хранить `score_present` и поверхность (`surface`), чтобы можно было строить отчёты.
- **Сравнение качества:** сделать A/B (если успеваете) **или** минимум «до/после» на одной аудитории.

Базовые метрики:

- **CTR (Click-through rate):** `click / impression`.
- **Play-through rate:** `play_start / impression`.
- **Skip rate:** `skip / play_start` (или `skip / impression` — зафиксировать один вариант в отчёте).

Критерий этапа: есть таблица/график метрик за период, описан baseline и итог после изменения scoring, вывод «улучшилось/не улучшилось» с цифрами.
