# Словарь данных MiMusic

Источник схемы: `mimusicback-master/db/schema/pgsql_starter_code.sql`, уточнения домена — `md/BACKEND_CLIENT_ALIGNMENT.md`.

Словарь данных описывает атрибуты каждой таблицы PostgreSQL, чтобы можно было понять, какие данные где хранятся, без изучения исходного кода.

---

## Таблица 1 — Словарь данных для сущности «Пользователи» (`users`)

| № | Атрибут | Описание атрибута |
|---|---------|-------------------|
| 1 | ID | Уникальный идентификатор пользователя |
| 2 | Электронная почта | Адрес электронной почты (логин), может быть пустым |
| 3 | Никнейм | Отображаемое имя; уникально в системе |
| 4 | Хэш пароля | Хэш пароля (не хранится открытый текст) |
| 5 | Ключ хранения аватара | Ключ файла аватара в `file_storage/avatars/` |
| 6 | Описание профиля | Текст описания профиля |
| 7 | Время создания | Время создания учётной записи |
| 8 | Время изменения | Время последнего изменения профиля |
| 9 | Признак администратора | Признак администратора платформы |

Таблица хранит учётные записи пользователей MiMusic. Через неё связаны треки, плейлисты, альбомы, социальные действия и сессии входа.

## Таблица 2 — Словарь данных для сущности «Треки» (`tracks`)

| № | Атрибут | Описание атрибута |
|---|---------|-------------------|
| 1 | ID | Уникальный идентификатор трека |
| 2 | ID загрузившего пользователя | Идентификатор пользователя, опубликовавшего трек |
| 3 | Название | Название трека |
| 4 | Исполнители | Массив имён исполнителей |
| 5 | Ключ хранения аудио | Ключ аудиофайла в `music_storage/` |
| 6 | Ключ хранения обложки | Ключ обложки в `file_storage/covers/tracks/` |
| 7 | Хэш файла | Хэш файла для дедупликации и проверки целостности |
| 8 | Длительность (мс) | Длительность трека в миллисекундах |
| 9 | Время создания | Время загрузки или создания записи |
| 10 | Время изменения | Время последнего изменения метаданных |

Единая медиатека приложения: любой авторизованный пользователь может воспроизводить трек, лайкать его и добавлять в свои плейлисты. В профиле автора отображаются все треки, где он указан как `uploader_user_id`.

## Связи между таблицами (кратко)

| Таблица | Связь |
|---------|--------|
| `tracks` | → `users` (uploader) |
| `albums`, `playlists` | → `users` (владелец) |
| `thoughts`, `comments` | → `users` (автор); `comments` → `thoughts` |
| `playlist_tracks` | `playlists` ↔ `tracks` |
| `album_tracks` | `albums` ↔ `tracks` |
| `track_likes`, `playlist_likes`, `thought_likes` | `users` ↔ сущность |
| `friend_requests`, `friendships` | `users` ↔ `users` |
| `colisten_sessions` | → `users` (host), → `tracks` (current); очередь и участники — отдельные таблицы |
| `notifications` | → `users` (recipient, actor) |
| `auth_sessions`, `listen_events`, `user_playback_queue_items`, `user_now_playing`, `user_presence` | → `users`; при необходимости → `tracks` |
| `invite_keys` | → `users` (creator) |
| `track_genres`, `album_genres` | сущность ↔ `genres` |
| `user_genre_preferences` | `users` ↔ `genres` |
| `recommendation_events` | → `users`; `target_id` без жёсткого FK |
