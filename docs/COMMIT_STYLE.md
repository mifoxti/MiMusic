# Как оформлять коммиты (в т.ч. при смене агента)

Цель: один коммит = один смысловой срез, сообщение читается человеком и по истории понятно, **что** сделали и **где** трогали код. При передаче задачи другому агенту достаточно заголовка + тела коммита, чтобы продолжить без догадок.

---

## 1. Заголовок (первая строка)

- **Тип коммита** — **на английском**, в стиле [Conventional Commits](https://www.conventionalcommits.org/): `feat`, `fix`, `refactor`, `chore`, `docs`, `style`, `test`, `perf` и т.д.
- **Область (scope)** — по желанию, **на английском**, в скобках: `feat(profile):`, `fix(player):`, `chore(l10n):`.
- **Описание после двоеточия** — **на русском**, **от первого лица** (прошедшее или совершенное: *добавил*, *исправил*, *вынес*, *подключил*, *обновил*). Без точки в конце привычно, но можно с точкой — главное единообразие в репозитории.

**Идеальный вид заголовка:**

```text
feat(profile): добавил файл аватара, обновил экран профиля и сделал надёжный выбор медиа
```

или без scope:

```text
feat: добавил файл аватара, обновил экран профиля и сделал надёжный выбор медиа
```

**Плохо:**

- `feat: profile avatar` — описание не по-русски и не от первого лица.
- `Добавил профиль` — нет английского типа.
- `feat(profile): Added avatar` — описание не на русском.

---

## 2. Тело коммита (после пустой строки)

Необязательно, но **желательно** для нетривиальных изменений и **обязательно удобно при смене агента**.

Правила:

1. **Пути к файлам, имена классов, функций, ключи l10n, эндпоинты** — **на английском**, как в коде (`lib/...`, `UserAvatar`, `GET /me`, `profile.edit.*`).
2. **Пояснения смысла** — **на русском**, можно от первого лица или нейтрально («сделал так-то»).
3. Маркеры списка: `-` по одному пункту на строку.
4. Логические блоки — группами: сначала UI, потом данные, потом платформа и т.д.

**Пример в духе «идеала»:**

```text
feat(profile): добавил файл аватара, обновил экран профиля и сделал надёжный выбор медиа

- Profile edit UI: field labels, cards, custom photo + presets, l10n keys profile.edit.*

- UserAvatar + profile screen: asset or file path; after settings save reload AppSettings and bump generation for image cache

- cover_pick_save: UI prep before picker, clearTemporaryFiles, readAsBytes for gallery; pickAndSaveTrackAudio + saveAudioBytesToApp

- Studio: audio pick via shared pipeline; no redundant copy when picking from gallery
```

Так и человек, и агент видят: **какие зоны кода** (англ.) и **зачем** (рус. смысл в заголовке + в буллетах).

---

## 3. Один коммит — один замысел

- Не смешивай в одном коммите «починил баг плеера» и «переименовал пакет» — два коммита.
- Крупную фичу можно разбить: `feat(api): …` затем `feat(home): …` — если каждый коммит собирается и осмыслен сам по себе.

---

## 4. Репозитории MiMusic

| Репозиторий | Пример ветки | Заметка |
|-------------|----------------|---------|
| `MiMusic/` (Flutter) | `master` | Коммиты только клиента. |
| `mimusicback-master/` (Ktor) | `main` / `master` | Коммиты только бэка. |

Документация в **`md/`** (этот файл, роадмап, alignment) может жить **вне** этих git-репозиториев — тогда коммиты в бэк/фронт **не** включают `md/` автоматически; либо перенеси `md` внутрь одного репо, либо веди отдельный git на уровень выше.

---

## 5. Шаблон для копирования

```text
feat(scope): кратко на русском от первого лица что сделал

- path/or/Component: what changed (English identifiers; Russian sense optional)

- another/file.dart: …
```

```text
fix(player): исправил зависание при переключении очереди с серверного трека

- lib/core/audio/mimusic_audio_handler.dart: itemId in queue map, MediaItem id stable for likes

- lib/features/home/presentation/pages/home_page.dart: same-track check via assetPath
```

---

## 6. Для агента при handoff

В конце сессии, если правок много: кратко перечисли коммиты или сделай один коммит с телом по шаблону выше — следующему агенту не придётся восстанавливать контекст только из `git diff`.
